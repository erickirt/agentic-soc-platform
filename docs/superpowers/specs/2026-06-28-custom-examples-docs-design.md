# Custom Examples Documentation Design

## Context

The current development documentation explains Mock Data, SIEM YAML, Module development, and Playbook development as separate topics. However, several source-tree examples are meant to work together:

- `backend/mock/siem/` generates SIEM-like mock logs.
- `backend/custom/data/siem/` contains SIEM YAML files that describe the mock indexes.
- `backend/custom/modules/` contains example Modules that convert raw alerts into ASP Cases, Alerts, and Artifacts.
- `backend/custom/data/modules/` contains raw alert samples for those Modules.
- `backend/custom/playbooks/` contains two custom Playbook examples.
- `backend/custom/data/playbooks/` contains the prompt files used by the Case Summary Playbook.

The documentation should make this relationship explicit so users understand that these files are not isolated examples. They form a reusable demonstration set for customization, SIEM field documentation, Module ingestion, and Playbook automation.

## Goals

1. Add a clear documentation entry point for the example assets under Development.
2. Explain how Mock SIEM logs, SIEM YAML, Modules, raw alert samples, and custom Playbooks relate.
3. Provide a place to document Splunk SPL and ELK ES|QL alert queries for the mock logs when the user supplies them.
4. Keep the existing focused pages for Mock Data, SIEM YAML, Module development, and Playbook development.
5. Add cross-links from the existing pages into the new example pages.
6. Document both source-tree usage and the option to copy examples into a Compose deployment `custom/` directory.
7. Keep Chinese and English documentation structurally aligned.

## Non-goals

1. Do not change backend example code.
2. Do not add screenshots or image placeholders.
3. Do not claim every SIEM YAML has a matching Module.
4. Do not claim the Mail Module is generated by the SIEM Mock log generator.
5. Do not run VitePress build unless explicitly requested.

## Page structure

Add a new Custom Examples area under the Development / 定制开发 sidebar:

```text
custom-examples/
custom-examples/siem-module-flow/
custom-examples/playbooks/
```

Recommended sidebar placement: after Custom Console and before the single-topic development guides, or as a small group near Module Development, Playbook Development, SIEM YAML, and Mock Data.

The overall Development section should use this information architecture:

- Development overview: explain the extension model and recommended reading path.
- Custom Console: explain runtime observation and validation.
- Module Development, Playbook Development, SIEM YAML, and Mock Data: remain focused single-topic guides.
- Custom Examples: act as the cookbook / end-to-end example area that shows how the single-topic pieces work together.

This keeps concepts and APIs in the existing pages while moving cross-cutting examples into a dedicated area.

### `custom-examples/`

Purpose: overview page.

Content:

- Explain that `backend/custom/` in the source tree is a test/example asset set.
- Explain that the release package `custom/` directory is an empty template by default.
- Explain that users can copy selected examples into a Compose deployment if they want to run them there.
- Show the high-level relationship:
  - Mock SIEM logs create test log data.
  - SIEM YAML describes indexes and fields.
  - Modules consume Redis Stream raw alerts and create ASP records.
  - Playbooks operate on Cases after ingestion.

### `custom-examples/siem-module-flow/`

Purpose: show how mock logs, SIEM YAML, raw alerts, and Modules fit together.

Required relationship table:

| Asset | Role | Relationship |
| --- | --- | --- |
| `siem-host-events.yaml` | SIEM index schema | Matches Host mock events and supports the EDR vssadmin scenario. |
| `edr_vssadmin_delete_shadows.py` | Module | Converts vssadmin-like raw alerts into ransomware investigation Cases. |
| `siem-aws-cloudtrail.yaml` | SIEM index schema | Matches CloudTrail mock events and supports the AttachUserPolicy scenario. |
| `aws_iam_privilege_escalation_attach_user_policy.py` | Module | Converts AttachUserPolicy-like raw alerts into IAM privilege escalation Cases. |
| `siem-network-traffic.yaml` | SIEM index schema | Supports network traffic and brute-force mock logs for SIEM query/rule examples; no dedicated Module currently exists. |
| `mail_user_report_phishing.py` | Module | Uses raw alert samples from `backend/custom/data/modules/`; it is not generated by the SIEM Mock log generator. |

Also document:

- Source-tree paths.
- Compose-copy paths.
- Optional Splunk SPL and ELK ES|QL alert queries for each scenario when available.
- Refresh/Validate workflow in Custom Console.
- Which Worker must be running for Modules.
- What outputs to expect: Case, Alert, Artifact, Enrichment, and AI analysis scheduling where applicable.

If the user provides concrete alert queries, include them near the matching scenario:

- Host vssadmin scenario: Splunk SPL and/or ELK ES|QL that finds `vssadmin.exe delete shadows`.
- AWS AttachUserPolicy scenario: Splunk SPL and/or ELK ES|QL that finds high-risk `AttachUserPolicy` activity.
- Network traffic/brute-force scenario: Splunk SPL and/or ELK ES|QL that finds suspicious network or authentication patterns.

Make it clear that these rules select evidence from Mock SIEM logs. The resulting alert/action integration must write to the Redis Stream name consumed by the matching Module when the scenario has a Module.

Provided Splunk SPL for the network brute-force / failed-then-success login scenario:

```spl
index=siem-network-traffic event.category=authentication (event.action=login_failed OR
  event.action=login_success)
  | search
      [search index=siem-network-traffic event.category=authentication event.action=login_failed
       | stats count AS failed_count BY source.ip, user.name
       | where failed_count >= 5
       | join source.ip, user.name
           [search index=siem-network-traffic event.category=authentication event.action=login_success
            | stats count AS success_count BY source.ip, user.name]
       | fields source.ip, user.name]
```

Provided ELK ES|QL for the AWS AttachUserPolicy high-risk success scenario:

```esql
FROM siem-aws-cloudtrail
  | WHERE event.action == "AttachUserPolicy"
  | WHERE event.risk_score > 80
  | WHERE event.outcome == "success"
  | WHERE
      requestParameters.policyArn IN
        (
          "arn:aws:iam::aws:policy/AdministratorAccess",
          "arn:aws:iam::aws:policy/IAMFullAccess"
        )
  | SORT @timestamp DESC
```

Provided ELK ES|QL for the Host vssadmin delete shadows scenario:

```esql
FROM siem-host-events
  | WHERE process.name == "vssadmin.exe"
  | WHERE risk_score >= 80
  | WHERE process.command_line LIKE "*delete*shadows*"
  | SORT @timestamp DESC
```

### `custom-examples/playbooks/`

Purpose: introduce the two custom Playbook examples.

Required sections:

- `case_summary.py`
  - Reads `custom/data/playbooks/case_summary/System_zh.md` or `System_en.md`.
  - Uses Runtime prompt language to select the prompt.
  - Calls the LLM and writes the result to the Case Summary field.
- `cmdb_enrichment.py`
  - Iterates Artifacts linked to the Case.
  - Queries CMDB context through the integration layer.
  - Writes results to Artifact Enrichment.

Also document:

- Source-tree paths.
- Compose-copy paths.
- Refresh/Validate workflow in Custom Console.
- Required Worker.
- Expected result locations in the ASP UI.
